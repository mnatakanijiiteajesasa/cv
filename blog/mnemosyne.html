Mnemosyne: A Graph Neural Network Memory for Persistent AI Agents

2026 Global AI Hackathon Series with Qwen FastAPI · PyTorch Geometric · GraphSAGE

Mnemosyne is the goddess of memory in Greek mythology, the mother of the nine Muses, the keeper of all that is known and remembered. It's also the name of a memory system I built for AI agents — one that scores what to remember using a graph neural network instead of similarity search alone.

Inspiration

From the competition brief, I was given a list of problems to choose from. I read through all of them. Persistent memory was the one I kept coming back to — not because it looked easy, but because it felt like the most honest problem on the list. Every other capability you might build into an agent runs into the same wall eventually: the agent doesn't remember you. It doesn't know what you told it last week, what you prefer, what you're working on, or what you already tried. You rebuild context from scratch every session, and the agent stays a stranger no matter how many times you talk to it.

I wanted to build something that solved that at a level deeper than storing a transcript and replaying it — a system where memory has structure, weight, and intelligence, where the agent doesn't just recall what you said, but understands what matters, what's related, and what can safely be forgotten.

The architectural idea came from an unexpected direction. I'd spent a lot of time watching how platforms like Twitter build personalized timelines from a user's very first interaction, and what struck me was that the recommendation engine isn't just matching content to stated preferences — it's learning a graph of relationships between topics, accounts, and engagement patterns. The platform is reasoning over a graph of behavior, not a flat list of likes.

That observation turned into a direct question: if an agent's memories are nodes, and semantic similarity creates edges between them, could a graph neural network learn which memories actually matter for a given query better than vector search alone? Mnemosyne was built to test that hypothesis.

What I Learned

Persistent memory in production is a largely unsolved problem. Before writing a line of code I researched how memory is actually handled in AI applications today. Most systems either append the full conversation history to every prompt — which hits context limits fast and buries what matters in noise — or keep a rolling summary that loses specificity over time. Neither gives an agent genuine long-term memory.

Not all memories are the same kind of thing. A behavioral rule ("always give concise answers"), a stable fact ("the user is a software engineer"), a stated preference ("the user prefers dark mode"), and a past episode ("the user asked about async IO last Tuesday") have different lifespans, different importance, and different roles in how an agent should reason. Building that taxonomy into the schema from day one, rather than treating all memories as undifferentiated text, was one of the most consequential early decisions in the project.

LangChain and LangGraph exist, and I chose not to use them. Both are widely used and both include memory abstractions — conversation buffer memory, summary memory, vector-store-backed memory, stateful agent loops. I studied them carefully. What I found was that these are convenience wrappers over three simple patterns: append history, summarize it, or retrieve by similarity. None of them offer typed memory, semantic graph structure, learned relevance scoring, or principled forgetting. They're useful for shipping fast. They aren't memory infrastructure. I built from first principles because the problem required it, and because I wanted to understand every component rather than inherit behavior I couldn't see into.

GNNs hadn't been applied to agent memory this way before. Before committing to GraphSAGE I searched extensively for prior work on graph neural networks applied specifically to persistent AI agent memory retrieval. I found knowledge graphs for question answering, graph-based recommendation systems, and GNNs in NLP generally — but nothing that applied learned graph-based scoring to personal memory retrieval for a conversational agent. That absence was both a risk and an opportunity: no playbook to follow, but a real shot at something novel if the hypothesis held.

The social media analogy runs deeper than it first appears. Both systems face the same core challenge — given a large, growing set of items connected by relational structure, rank the ones most relevant to this user right now. Twitter solves it with a graph of engagement signals. Mnemosyne solves it with a graph of semantic similarity and access patterns, scored by a GNN trained on relevance labels. Even the cold-start problem is structurally identical: a new user has no history, so the system can't yet personalize, and has to degrade gracefully to a safe default while real signal accumulates.

Graceful degradation is a design discipline. Every learned component — the GNN relevance scores, the cluster confidence — has to fall back to a safe baseline when data is sparse. A system that fails hard at cold start isn't a memory system, it's a liability. Designing a confidence gate so the GNN always runs but its output is only trusted in proportion to the evidence behind it was one of the most important architectural calls in the project.

Memory is a UX problem as much as a technical one. A system that generates a fresh random user ID every session structurally prevents memory from accumulating, no matter how well the rest of the pipeline works. The single most impactful change in the entire project was replacing dynamic ID generation with a fixed, persistent identifier. Architecture is nothing without coherent identity across sessions.

How I Built It

Mnemosyne runs a four-layer pipeline over every conversational turn, end to end.

Memory schema

Every piece of information the agent learns is classified into one of four typed memory categories, each with a distinct role and a base importance score that gets adjusted at write time with a keyword boost:

TypeRoleBase importance
RULEBehavioral directives the agent must follow0.85
PREFERENCEHow the user likes things done0.70
FACTStable information about the user0.60
EPISODESpecific past events0.40

The memory graph

After every write, the new memory is compared against the user's existing memories in Qdrant. Any pair whose cosine similarity clears a fixed threshold gets a weighted edge in the graph. This produces semantic clusters — work memories connect to other work memories, preferences cluster with preferences — giving the GNN a structured graph to reason over rather than an unconnected set of points.

The GraphSAGE model

Each memory node is represented as a 391-dimensional feature vector, concatenating:

x_i = [ e_i (384-dim sentence embedding) | t_i (4-dim one-hot memory type) | s_i (importance score) | a_i (normalized age) | c_i (normalized access count) ]

Two GraphSAGE layers with mean aggregation produce enriched node embeddings, where each layer combines a node's own representation with the mean of its neighbors':

h_i^(l+1) = σ( W^(l) · CONCAT( h_i^(l), MEAN_{j ∈ N(i)} h_j^(l) ) )

The model has two parallel output heads:

  • Relevance head — a sigmoid MLP predicting whether a memory will be accessed, output in [0, 1].
  • Cluster head — a softmax MLP predicting memory type from neighborhood structure, output in ℝ⁴.

Training uses a combined loss with dual supervision:

L = BCE(r_i, y_i^rel) + α · CE(c_i, y_i^type), α = 0.5

The cluster head acts as a regularizer, stopping the shared embedding from collapsing to shortcuts that ignore memory-type structure.

Hybrid retrieval

At inference time, four signals are blended into a single score per candidate memory: cosine similarity from Qdrant, GNN relevance from GraphSAGE, recency (which decays smoothly with turns since last access), and cluster confidence from the softmax head. Results are cached in Redis with a 30-second TTL — short enough that memory state changes show up quickly, long enough to absorb repeated queries within a turn.

Memory aging and forgetting

Every turn increments a turns_since_access counter for each of the user's active memories; accessing a memory resets its counter to zero. Every ten turns, a forgetting pass archives memories whose importance and recency have both decayed below threshold. Archived memories are never deleted, only excluded from active retrieval — preserving the option to recover or audit history without losing data.

The LLM integration

Retrieved memories are grouped by type and formatted into a natural-language context block prepended to the system prompt sent to Qwen via Alibaba Cloud Model Studio. The prompt explicitly instructs the model to treat RULE memories as binding behavioral instructions and everything else as context — a distinction that matters enormously in practice and that generic memory systems typically ignore.

Challenges

The user identity problem. The single most damaging bug in the project had nothing to do with machine learning. Generating a fresh random user ID per session meant the system was structurally incapable of accumulating memory, no matter how well every other component worked. Memory persistence requires persistent identity — obvious in retrospect, invisible until you step back from the code and ask why the graph never grows.

Silent payload omissions. The memory-write pipeline stored embeddings in Qdrant without including the raw text content in the payload. Retrieval returned correct memory IDs and similarity scores. The LLM received empty strings for every memory context. No error was raised anywhere in the pipeline. This class of bug — where the system is correct by every measurable metric but wrong at the boundary between components — is the hardest to find and the most important to design against.

The cold-start problem. A new user has no memory graph. A sparse graph produces weak GNN signal. Weak GNN signal means the system runs on flat vector retrieval — no better than a basic RAG system. The fix is a confidence gate that blends GNN output with flat retrieval in proportion to how much real evidence exists, so the GNN runs from turn one but its output is only trusted as far as the evidence supports. The transition from cold start to full personalization is a smooth handoff rather than a threshold the user can feel.

Deployment. First deployment to a real server on Alibaba Cloud ECS surfaced everything local Docker Compose doesn't: SSH key configuration, security group rules, the difference between a named Docker service hostname and localhost, and the fact that every port exposed in docker-compose.yml also needs opening in the cloud firewall independently. None of these are hard problems individually. All of them together, for the first time, on a deadline, are a real challenge.

Accomplishments

  • Applying GNNs to a problem where they hadn't been applied before — GraphSAGE for personal AI agent memory retrieval, built and validated as a deliberate hypothesis rather than an obvious default.
  • A working GraphSAGE model integrated into a live inference pipeline, not a notebook experiment — it runs inside a containerized FastAPI app on every retrieval call and feeds directly into the hybrid scoring formula.
  • A multi-database architecture where every store serves a distinct access pattern: MongoDB for canonical records and graph edges, Qdrant for filtered vector search, Redis for short-TTL retrieval caching.
  • A retrieval pipeline that blends four independent, weighted signals into a single learned score — a real departure from ranking by similarity alone.
  • A principled forgetting mechanism that archives rather than deletes, tied automatically to the session turn counter.

What's Next

GNN training from real interaction data. The model currently initializes with random weights and a heuristic relevance proxy for supervision. As Mnemosyne accumulates real usage — which memories get retrieved, which lead to good responses, which get accessed repeatedly — those become genuine training labels for a background worker that continuously fine-tunes the GNN.

Multi-user scaling. The architecture is correct for multi-user use but has only been validated with a single persistent user. Real multi-user deployment needs per-user graph isolation at scale, more aggressive caching, and background aging/forgetting workers that don't block the main request path.

Memory export and portability. A user's memory graph should belong to them. The next version adds an export mechanism — a structured format that lets someone take their memory out of the system, inspect it, or import it elsewhere.

The long-term vision is for Mnemosyne to become the memory layer any AI agent can be built on — not a product, but infrastructure. The same way a database isn't an application, Mnemosyne isn't an assistant. It's what makes an assistant worth coming back to.
